# ShellPilot 官方网站与内容后台设计

**日期：** 2026-07-18

**状态：** 产品与技术方案已由用户确认

**范围：** 官方网站、内容管理后台、静态发布服务、版本同步、匿名留言，以及默认禁用的商业后台骨架

## 1. 结论

ShellPilot 官方网站采用“可扩展静态官网 + 完整内容管理后台 + 后期独立商业能力”的结构。

公开网站使用 Astro、React 和 TypeScript 构建为静态文件。内容后台使用 Payload CMS 和 PostgreSQL，提供单管理员登录、内容编辑、草稿、预览、素材管理、发布历史和回滚。管理员点击发布后，独立发布任务从后台导出已发布内容，构建并验证新的静态网站，验证通过后再原子切换线上目录。

软件版本信息与现有 GitHub Releases、ModelScope 发布流程连接。留言允许访客免登录提交，但默认先审核后公开。商业模块只建立代码边界、数据模型和关闭状态下的接口行为，不接真实支付，不开放公众账号、订单或套餐购买入口。

该方案避免从零开发通用 CMS 能力，同时保持公开网站的品牌定制、静态性能、故障隔离和未来扩展能力。

## 2. 已确认的产品决策

1. 第一阶段建设公开官网和完整内容管理后台。
2. 公开官网采用“宣传首页 + 独立内容页”，不把所有内容堆在单个长页面。
3. 后台采用“编辑草稿、完整预览、点击发布、生成静态官网”的发布模式。
4. 初期只有一个超级管理员账号，数据结构预留管理员、编辑和审核员角色。
5. 访客不注册、不登录即可留言；管理员必须登录后台后才能审核和回复。
6. 最新版本、下载链接、文件大小、校验值和更新内容自动同步。
7. 官网内容发布失败时不得影响当前线上版本。
8. 保留最近 10 个成功发布的网站快照，并支持一键回滚。
9. 内容后台完整可用；商业后台只建设模块骨架和接口边界，默认关闭。
10. 当前工作只设计网站，不修改 ShellPilot 客户端功能，也不设计客户端授权或激活机制。

## 3. 目标

- 建立可信、清晰、适合工具产品的官方下载入口。
- 准确展示 ShellPilot 已经发布的能力，不宣传未交付功能。
- 让站长不修改代码即可维护首页、截图、文档、FAQ、更新日志和联系信息。
- 让软件新版本发布后，官网的版本和下载信息自动更新。
- 让公开官网在后台、数据库或留言服务故障时仍可访问。
- 为赞赏、版本套餐、用户、订单和支付留下稳定的扩展边界。
- 支持在现有美国 VPS 自托管。
- 在尚未购买域名时支持通过 IP 完成预览和验收。

## 4. 非目标

- 第一阶段不提供公众注册、登录或用户中心。
- 第一阶段不接入真实支付、退款、优惠券核销或订单履约。
- 第一阶段不开放普通版与高级版购买入口。
- 第一阶段不建设客户端授权、设备绑定或许可证验证。
- 安装包不存储在官网 VPS；下载继续使用 GitHub Releases 和 ModelScope。
- 第一阶段不建设多人邀请、公开密码找回或复杂组织权限。
- 第一阶段不建设多语言站点。
- 本规格不决定最终视觉主题。视觉方向在前端实施前从现有预览方案中单独确认，不改变本规格的页面结构和数据模型。

## 5. 方案比较与选择

### 5.1 选定方案：Astro + Payload CMS + PostgreSQL

- Astro 负责公开静态站点、内容页面、文档路由、搜索索引和构建期校验。
- React 只用于截图浏览、FAQ 展开、搜索、留言表单等需要交互的局部组件。
- Payload CMS 提供管理界面、认证、权限、素材上传、草稿和内容版本。
- PostgreSQL 保存后台内容、版本、留言、审计和发布任务。
- 发布服务把 Payload 中已发布的内容导出为固定快照，再构建 Astro 静态文件。
- Caddy 提供公开静态文件并反向代理必要的后台与留言接口。

这一组合保留 TypeScript 和 React 生态，同时避免从零开发 CMS 的通用能力。

### 5.2 未选择：React/Vite 单页站 + 完全自建后台

该方案与当前客户端栈最接近，但文档静态生成、内容版本、草稿、素材库、权限和后台表单都需要自行建设。首期成本更高，且容易把网站后台与商业逻辑耦合。

### 5.3 未选择：WordPress 或完整主题站

该方案上线快，但需要维护 PHP、主题和插件生态，产品页面定制、现有版本发布流程连接，以及未来独立商业服务的边界都不够自然。

## 6. 系统边界

系统由五个相互隔离的部分组成：

1. **公开静态官网：** 只读取构建时生成的内容，不依赖后台运行时响应。
2. **内容管理后台：** 管理内容、素材、版本、留言和发布任务。
3. **发布服务：** 导出快照、验证数据、构建网站、运行检查、切换版本和回滚。
4. **动态接口：** 只承载匿名留言提交、管理员操作和版本同步通知。
5. **商业模块骨架：** 默认关闭，不注册公开路由，不参与第一阶段数据流。

数据流如下：

    管理员编辑内容
        -> Payload 保存草稿与内容版本
        -> 管理员打开完整预览
        -> 管理员点击发布
        -> 发布服务导出不可变内容快照
        -> Astro 构建静态站点
        -> 自动检查页面、资源和下载链接
        -> 原子切换 Caddy 当前站点目录
        -> 保留旧快照供回滚

后台、数据库或发布服务中断时，Caddy 继续提供最后一次成功发布的静态目录。

## 7. 公开官网页面

### 7.1 首页

首页包含：

- 产品名称、简短定位和主视觉。
- 主要下载按钮与备用下载入口。
- 当前稳定版本、发布日期和支持平台。
- 核心功能分组展示。
- 真实产品截图与可访问的截图说明。
- 安全、稳定性和开源来源说明。
- 常见问题摘要。
- 文档、更新日志和留言入口。
- 关于、联系、隐私和许可页脚链接。

首页后台支持修改各模块文案、图片、按钮、顺序、显示状态和 SEO 信息。

### 7.2 下载中心

下载中心展示：

- 当前稳定版本。
- 安装包名称、架构、文件大小和 SHA256。
- GitHub Releases 主下载地址。
- ModelScope 备用下载地址；同步未完成时不显示失效入口。
- 发布日期、更新摘要和完整更新日志链接。
- 系统要求和安装提示。
- 历史稳定版本列表。

网站不代理安装包，不把下载流量转发到 VPS。

### 7.3 使用文档

文档支持：

- 分类与层级目录。
- 页面内目录。
- 关键词搜索。
- 上一篇和下一篇导航。
- 代码块、提示框、图片和外部链接。
- 文章更新时间。
- 指向留言反馈的“文档是否有帮助”入口。

搜索索引在发布时生成并随静态网站部署，不依赖运行时搜索服务。

### 7.4 更新日志

- 按版本和发布日期倒序排列。
- 区分新增、改进、修复和已知问题。
- 与下载中心的版本记录关联。
- 支持后台手工补充说明，但版本号、安装包和校验值以发布同步数据为准。

### 7.5 FAQ

- 支持分类、关键词搜索和折叠查看。
- 支持后台排序、隐藏和重新发布。
- 首页只展示后台指定的精选问题。

### 7.6 留言反馈

- 访客填写昵称和留言内容。
- 联系方式可选，永不在公开页面展示。
- 提交后明确提示“等待审核”，不伪装成已公开。
- 审核通过的留言和管理员公开回复可见。
- 留言接口不可用时，显示暂时不可用状态，不影响其他页面。

### 7.7 关于、联系与法律页面

- 关于页面介绍项目背景、维护方式和发布渠道。
- 联系页面由后台配置邮箱、GitHub、交流群或其他公开渠道。
- 隐私声明说明留言收集、日志、Cookie 和联系方式的使用范围。
- 开源许可页面保存 ShellPilot 及其上游项目所需的许可和归属说明。

## 8. 内容管理后台

### 8.1 登录与权限

- 第一阶段只有一个超级管理员。
- 管理员可编辑、预览、发布、回滚、管理素材和审核留言。
- 用户表保留角色字段，但不提供邀请、编辑员或审核员创建界面。
- 不提供公开密码找回；初始账号通过部署命令创建，管理员登录后可修改密码。
- 连续登录失败触发速率限制和临时锁定。
- 所有会改变公开内容的操作进入审计日志。

### 8.2 控制台

控制台显示：

- 当前线上网站版本。
- 未发布草稿数量。
- 最近发布任务及结果。
- 待审核留言数量。
- 当前软件稳定版本与同步状态。
- 数据库最近备份时间。
- 商业模块关闭状态。

### 8.3 内容与素材

后台管理以下集合：

- 站点设置、导航和页脚。
- 首页模块。
- 功能展示与截图。
- 文档分类和文档文章。
- FAQ 分类和问题。
- 软件发布、更新日志和下载源。
- 关于、联系、隐私和许可页面。
- 图片与附件素材。
- 匿名留言和管理员回复。

内容支持草稿、预览、发布状态、排序、定时字段预留和历史版本恢复。

### 8.4 发布中心

发布中心支持：

- 查看本次待发布变更。
- 打开使用草稿数据生成的完整预览。
- 发起单个发布任务。
- 查看构建和检查日志。
- 查看最近 10 个成功网站版本。
- 将当前网站切换回任意保留版本。
- 禁止并行运行两个发布或回滚任务。

## 9. 核心数据模型

### 9.1 内容数据

- SiteSettings：站点名称、描述、公共地址、默认分享图和功能开关。
- NavigationItem：导航名称、目标、顺序和显示状态。
- HomeSection：首页区块类型、内容、顺序和显示状态。
- FeatureItem：功能名称、说明、图标和截图关联。
- Screenshot：图片、替代文本、说明和排序。
- DocCategory：文档分类、父级和排序。
- DocArticle：标题、路径、摘要、正文、分类和发布状态。
- FaqItem：问题、答案、分类、精选状态和排序。
- Release：版本号、发布日期、状态、更新摘要和来源标识。
- ReleaseAsset：平台、架构、文件名、大小、SHA256、主地址和镜像地址。
- LegalPage：页面类型、标题、正文和更新时间。
- ContactLink：类型、名称、地址、公开状态和排序。
- Media：文件、类型、尺寸、替代文本和引用状态。

### 9.2 运行数据

- AdminUser：管理员身份、密码摘要、角色和锁定状态。
- GuestMessage：昵称、私密联系方式、正文、审核状态和反垃圾元数据。
- GuestReply：留言关联、回复正文和公开状态。
- PublishJob：快照 ID、状态、日志、开始时间、结束时间和失败原因。
- SiteDeployment：发布目录、内容摘要、构建摘要和当前状态。
- AuditEvent：管理员、操作、对象、结果、时间和请求摘要。
- ReleaseSyncEvent：发布来源、签名校验结果、导入结果和错误。

敏感联系方式不进入公开快照。反垃圾 IP 只保存带轮换盐的摘要，不保存可公开查询的明文地址。

## 10. 软件版本自动同步

现有软件发布流程在 GitHub Release 成功后调用网站的签名同步接口。

同步接口执行：

1. 校验 HMAC 签名、时间戳和重放标识。
2. 校验版本号与发布来源。
3. 获取安装包元数据、文件大小和 SHA256。
4. 验证 GitHub 主下载地址可访问。
5. 尝试验证 ModelScope 镜像；镜像未就绪时记录告警但不发布失效链接。
6. 保存 Release 和 ReleaseAsset。
7. 自动触发一次仅包含版本内容的静态发布。

GitHub 主安装包、文件大小或 SHA256 不完整时，不替换官网当前稳定版本。ModelScope 暂未同步时，官网可以只显示 GitHub 下载，待镜像验证通过后再次发布。

同步接口必须幂等；同一版本和同一资源摘要的重复通知不重复创建版本或发布任务。

## 11. 匿名留言与审核

留言接口由后台服务提供，公开网站只提交必要字段。

防滥用规则包括：

- 昵称和正文长度限制。
- 服务端 HTML 清理与输出转义。
- 单来源速率限制。
- 相同内容短时间重复检测。
- 隐藏蜜罐字段。
- 反垃圾 IP 摘要。
- 后台手工标记垃圾和批量隐藏。

所有新留言默认状态为 pending。管理员可以批准、公开回复、隐藏、标记垃圾或软删除。软删除数据在 30 天后由清理任务永久删除；审计记录只保留对象 ID、操作和结果，不保留已删除正文。

## 12. 静态发布与回滚

### 12.1 发布步骤

1. 为已发布内容生成不可变快照和内容摘要。
2. 检查必填字段、唯一页面路径、素材引用和内部链接。
3. 在独立临时目录执行 Astro 构建。
4. 检查首页、下载、文档、FAQ、更新日志和法律页面。
5. 检查静态资源、搜索索引、站点地图和下载地址。
6. 将成功构建移动到新的版本目录。
7. 原子切换 current 指针。
8. 从外部地址执行最终 Smoke 检查。
9. 记录发布结果并清理超过 10 个的旧版本。

### 12.2 失败处理

- 数据校验失败：不开始构建，不改变线上版本。
- 构建失败：保留构建日志，不改变线上版本。
- 切换前检查失败：删除临时目录，不改变线上版本。
- 切换后 Smoke 失败：立即切回上一个成功版本并记录严重告警。
- 回滚失败：保持 current 指向最后可确认存在的完整目录，并禁止继续发布，直到管理员处理。

## 13. 商业后台骨架

商业骨架与内容集合分开组织，包含：

- Plans：套餐定义。
- Customers：未来客户记录。
- Orders：未来订单记录。
- Payments：未来支付记录。
- Discounts：未来优惠规则。
- CommercialEvents：未来商业事件审计。

第一阶段约束：

- 全局功能开关固定为关闭。
- 管理菜单不显示商业模块。
- 不注册面向公众的商业 API。
- 内部商业接口调用返回明确的 disabled 结果。
- 不配置支付密钥，不创建模拟交易，不运行订单定时任务。
- 内容发布流程不读取商业表。

未来启用商业能力时，先编写独立商业规格；本骨架只保证目录、数据迁移边界和接口命名稳定。

## 14. 部署设计

美国 VPS 使用 Docker Compose 运行：

- Payload CMS 服务。
- PostgreSQL。
- 发布服务。
- Caddy。

公开静态文件存放在主机持久目录，Caddy 只读取 current 指向的成功发布版本。媒体文件、数据库和备份使用独立持久卷。

### 14.1 无域名阶段

- 公开静态网站可以通过 VPS IP 提供预览和验收。
- 管理后台不通过明文 HTTP 暴露到公网，只绑定本机地址并通过 SSH 隧道访问。
- 留言可以在本地或受控测试环境验收；正式公开收集留言前配置域名和可信 HTTPS。

### 14.2 获得域名后

- 将域名解析到 VPS。
- 由 Caddy 申请并续期 TLS 证书。
- 后台使用安全 Cookie、可信站点来源和严格跨域规则。
- 配置 PUBLIC_SITE_URL 并重新发布，以生成正式 canonical、站点地图和分享链接。

## 15. 备份与保留

- PostgreSQL 每日备份，保留最近 14 个日备份和 8 个周备份。
- 媒体文件每日增量备份。
- 最近 10 个成功网站构建保留在 VPS。
- 发布前不依赖实时备份完成，但后台必须显示最近一次成功备份时间。
- 每次正式上线前执行一次数据库恢复和静态目录恢复演练。

## 16. 安全与隐私

- 后台和管理 API 默认拒绝未认证访问。
- 单管理员账号使用强密码、HttpOnly Cookie、SameSite 和可信 HTTPS。
- 登录、发布、回滚、留言删除和站点设置变更进入审计日志。
- 发布同步接口使用 HMAC、短有效期时间戳和防重放标识。
- Markdown 与富文本在保存和渲染两侧执行允许列表清理。
- 上传文件限制 MIME、扩展名和大小，图片重新解析后再公开。
- 后台数据库、备份和商业骨架接口不向公众开放。
- 留言联系方式仅管理员可见，不进入公开静态文件。
- 日志不得记录密码、Cookie、完整签名密钥或私密联系方式。

## 17. SEO、可访问性与性能

- 每个公开页面都有独立标题、描述、canonical 和分享图片。
- 构建时生成 sitemap.xml、robots.txt 和文档搜索索引。
- 图片必须包含替代文本并生成合理尺寸。
- 导航、FAQ、截图浏览和留言表单支持键盘操作。
- 页面在常见桌面和移动宽度下不产生横向滚动。
- 首屏不依赖后台 API；留言脚本失败不阻塞主体内容。
- 公开页面默认输出静态 HTML，交互组件按需加载。

## 18. 测试策略

### 18.1 单元测试

- 内容结构和路径唯一性校验。
- 版本号、文件大小、SHA256 和下载地址校验。
- 发布同步签名、时间戳、幂等和防重放。
- 留言字段、速率限制、重复检测和输出清理。
- 商业功能开关关闭时的拒绝行为。

### 18.2 集成测试

- Payload 草稿、预览、发布和内容版本恢复。
- 媒体上传与引用检查。
- 内容快照导出与 Astro 构建。
- 发布目录原子切换和回滚。
- GitHub 主下载成功、ModelScope 延迟和两个来源都失败的处理。
- 留言提交、审核、回复、隐藏和删除。
- 数据库备份与恢复。

### 18.3 E2E 测试

- 首页到下载中心的完整访问路径。
- 文档分类、搜索和上一篇/下一篇。
- FAQ 分类和搜索。
- 访客免登录留言及等待审核提示。
- 管理员登录、编辑首页、预览、发布和回滚。
- 移动端导航、键盘焦点和常见浏览器布局。

### 18.4 部署 Smoke

- 首页、下载中心、至少一篇文档、FAQ、更新日志和法律页面返回成功。
- 静态资源和搜索索引可读取。
- 当前稳定版本与后台记录一致。
- 主下载链接可访问且展示的校验值完整。
- 后台故障时公开静态页面仍可访问。
- 商业接口不可从公网访问。

## 19. 验收标准

1. 公开网站包含首页、下载、文档、更新日志、FAQ、留言、关于、联系、隐私和许可页面。
2. 站长无需修改代码即可维护所有第一阶段内容。
3. 后台支持草稿、完整预览、发布历史和最近 10 个版本回滚。
4. 发布校验或构建失败不会替换当前线上网站。
5. 软件发布成功后，版本、安装包、大小、SHA256、下载地址和更新内容能够自动同步。
6. GitHub 主下载不可用时不更新当前稳定版本。
7. ModelScope 尚未就绪时不展示失效镜像链接。
8. 访客无需账号即可留言，未审核留言不公开。
9. 后台或数据库停止时，最后一次成功发布的公开网站仍可访问。
10. 初期只有一个超级管理员，同时保留未来多角色数据边界。
11. 商业模块默认关闭，不显示菜单、不开放公众接口、不处理交易。
12. 无域名时可通过 IP 完成公开站点预览，后台通过 SSH 隧道安全访问。
13. 获得域名后可以在不改变应用架构的情况下启用 HTTPS 和正式公开留言。
14. 数据库、媒体和静态版本都具有明确的备份和恢复路径。

## 20. 建议实施顺序

1. 建立 Astro 公开站点、Payload CMS、PostgreSQL 和 Caddy 的工程骨架。
2. 建立内容模型、单管理员登录和素材库。
3. 完成首页、下载、文档、更新日志、FAQ、关于、联系和法律页面。
4. 完成快照导出、静态构建、发布记录、原子切换和回滚。
5. 接入软件版本同步。
6. 完成匿名留言和后台审核。
7. 建立默认禁用的商业模块骨架。
8. 完成安全、备份、E2E、恢复演练和 VPS 部署验收。
