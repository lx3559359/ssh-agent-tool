# ShellPilot AI 运维安全事务设计

**日期：** 2026-07-13
**目标版本：** 0.4.0
**状态：** 已确认方案 B，待实施计划

## 1. 目标

ShellPilot 将服务器状态、AI Agent、快捷命令、普通 SSH 终端和安全操作中心接入同一套安全事务引擎，实现以下闭环：

1. 从服务器异常生成一键诊断任务。
2. AI 先生成结构化只读排查计划，用户确认后执行。
3. 修改操作必须先创建可验证恢复点，再二次确认。
4. 所有步骤形成统一任务记录和脱敏审计报告。
5. 可恢复操作可以在安全操作中心点击按钮回滚。
6. 用户在普通 SSH 终端中手工输入受支持的修改命令时，也执行相同的备份和回滚策略。

本设计不承诺任意命令均可回滚。数据库写入、软件升级、任意脚本内部修改以及交互式程序内部操作无法通用恢复，只记录风险和审计信息。

## 2. 设计原则

- **SSH 底座不重写：** 继续复用 Electerm 的终端、SSH、SFTP、会话和命令执行能力。
- **单一安全入口：** AI、快捷命令、异常诊断和终端手工命令共用事务模型、风险分类、恢复点和审计记录。
- **恢复点失败即停止：** 声明可回滚的操作若无法创建并验证恢复点，不允许执行修改命令。
- **只读透明：** 已识别的只读命令不增加二次确认，不改变常规 SSH 使用习惯。
- **不虚假承诺：** 未知命令或不可逆操作显示明确警告，不显示“可回滚”。
- **最小敏感数据：** 不保存认证密码、私钥、API Key、Token；命令输出限长并进行内容级脱敏。
- **精确会话匹配：** 执行和回滚必须匹配原主机、端口、账号和 SSH 会话类型。

## 3. 总体架构

### 3.1 安全事务引擎

新增独立的 `safety-transaction` 领域模块，包含：

- `command-classifier`：识别只读、可恢复修改、高风险不可恢复和未知命令。
- `recovery-providers`：按文件、权限、systemd、防火墙、网络和容器生成恢复点及回滚动作。
- `transaction-runner`：串行执行准备、确认、修改、验证和回滚步骤，支持取消与超时。
- `transaction-records`：持久化任务、步骤、恢复点、输出摘要和状态迁移。
- `audit-redaction`：统一清理密码、Token、API Key、认证 Header 和私钥内容。
- `endpoint-guard`：执行前验证主机、端口、账号、标签页和终端 PID。

各调用方只提交统一请求：

```js
{
  source: 'terminal' | 'agent' | 'quick-command' | 'server-status',
  endpoint: { tabId, host, port, username },
  title,
  command,
  risk,
  recoveryProvider,
  metadata
}
```

调用方不得自行写入安全记录或拼接回滚命令。

### 3.2 远端恢复点

恢复点默认保存在当前 SSH 账号的：

```text
~/.shellpilot/operations/<operation-id>/
```

目录权限为 `700`，记录和备份文件权限为 `600`。目录包含：

- `manifest.json`：操作类型、目标、创建时间、校验摘要和原始状态。
- `rollback.sh`：只包含经过注册的回滚步骤。
- `verify.sh`：判断恢复点是否完整、目标是否仍适合恢复。
- `backup/`：原文件、权限或结构化状态数据。
- `result.json`：修改及验证结果，不包含凭据。

新事务不再依赖 `/tmp/shellpilot-rollback`。旧记录继续兼容读取和回滚，但不迁移远端文件。

## 4. 命令安全分类

### 4.1 只读命令

明确识别的查询命令直接执行，例如 `cat`、`less`、`tail`、`grep`、`find`、`ls`、`stat`、`df`、`du`、`free`、`ps`、`ss`、`ip address show`、`systemctl status/show/list-*`、`journalctl`、`docker ps/logs/inspect`。

命令链中只要包含重定向写入、命令替换后的未知执行、`eval`、`source`、脚本解释器执行或未识别子命令，就不能按只读处理。

### 4.2 首期可恢复修改

| 类型 | 识别范围 | 恢复点 | 回滚方式 |
| --- | --- | --- | --- |
| 文件修改 | `sed -i`、`tee`、重定向覆盖、`cp` 覆盖、已知编辑保存事件 | 原文件、权限、属主、摘要 | 原子恢复文件与元数据 |
| 文件删除 | `rm` 的明确文件或目录参数 | 移入当前账号回收区并记录原路径 | 移回原路径；冲突时停止 |
| 权限归属 | `chmod`、`chown`、`chgrp` | 原 mode、uid、gid | 恢复原权限和归属 |
| systemd | `start/stop/restart/enable/disable` | active、enabled 和 failed 状态 | 恢复运行及自启状态 |
| 防火墙 | firewalld、ufw 的端口增删 | 相关规则及 provider 状态 | 恢复相关规则并重新加载 |
| 网络 IP | `ip addr/route` 及已支持 NetworkManager 表单 | 地址、路由、DNS 和连接配置 | 定时自动回滚或手动回滚 |
| 容器状态 | Docker/Podman `start/stop/restart` | 原运行状态、容器 ID | 恢复原运行状态 |

所有命令必须先经过安全参数解析。包含通配符、未展开变量、危险根路径、跨文件系统不确定目标或无法唯一确定对象时，降级为高风险不可恢复。

### 4.3 高风险不可恢复

数据库写入、包管理升级、磁盘分区与格式化、内核参数持久化、用户密码修改、证书签发、任意脚本执行、`eval`、递归删除根级目录、Kubernetes 资源修改等首期不提供自动回滚。

这些命令默认弹出风险说明和二次确认，并写入审计记录。极高风险命令可以由设置项配置为直接阻止，但默认不改变普通 SSH 客户端对用户输入的控制权。

## 5. 普通 SSH 终端保护

### 5.1 拦截时机

在用户按 Enter、终端拥有完整当前命令且不处于密码输入、交互式程序、粘贴流或 heredoc 状态时分类命令。

- 只读命令：立即发送。
- 已支持的修改命令：暂停发送，显示安全确认面板。
- 高风险不可恢复命令：显示风险和“不可自动回滚”说明。
- 无法获得完整命令：保持终端原行为，同时只记录有限的命令完成事件，不宣称可回滚。

终端保护默认开启，可在设置中关闭。关闭后不拦截手工命令，但 AI 和快捷命令仍必须经过安全事务引擎。

### 5.2 文件编辑器限制

无法可靠观察 `vim`、`nano`、远端脚本或应用内部的每一次写入。首期仅在 ShellPilot 自带编辑器和 SFTP 编辑保存流程中创建文件恢复点；纯终端编辑器操作标记为“已审计、无自动恢复点”。

### 5.3 手工命令确认面板

面板显示：

- 命令与目标摘要。
- 风险等级及识别依据。
- 将创建的恢复点。
- 可恢复和不可恢复的范围。
- “创建恢复点并执行”“仅执行一次”“取消”三个明确动作。

网络修改不提供“仅执行一次”，必须启用自动回滚保护或取消。

## 6. 异常一键诊断

服务器状态中心的异常服务、异常资源和失败探针增加“AI 诊断”按钮。诊断请求只包含当前异常及必要关联数据：

- 服务 `systemctl show/status` 和最近日志。
- 服务进程、监听端口及网络关联。
- CPU、内存、磁盘和 inode 异常。
- 对应 Docker/Podman 容器信息。
- 已脱敏的服务器状态快照。

AI 不直接返回待执行的自由文本，而是返回结构化计划：

```js
{
  summary,
  steps: [{ id, title, command, purpose, readOnly, risk, timeoutMs }],
  expectedSignals,
  stopConditions
}
```

计划解析失败时仅显示 AI 文本，不提供自动执行按钮。

## 7. 排查计划执行

### 7.1 只读阶段

用户确认计划后，任务执行器按顺序运行只读步骤。界面显示等待、运行、成功、失败、跳过和已取消状态，并实时追加限长输出。

用户取消时：

1. 停止后续步骤。
2. 通过现有 SSH 命令超时/关闭能力终止当前命令流。
3. 已完成步骤保留在任务记录中。
4. 不把已取消任务标记为成功。

### 7.2 修复阶段

AI 根据只读结果提出修复建议，但不能自动执行。每个修改步骤必须转换成已注册安全事务；无法转换的步骤只允许复制到终端，并显示不可自动回滚警告。

可恢复修改的流程固定为：

```text
创建恢复点 → 验证恢复点 → 二次确认 → 执行修改 → 验证结果 → 等待保留或回滚
```

## 8. 状态机

任务状态：

```text
draft → awaiting-plan-confirmation → running-readonly
      → awaiting-change-confirmation → running-change
      → completed | failed | cancelled | partially-completed
```

安全事务状态：

```text
preparing → recovery-ready → awaiting-confirmation → executing
          → verification-passed → rollback-available
          → kept | rolling-back → restored
```

任意阶段都可以进入 `failed`。只有 `recovery-ready` 之后才允许修改；只有远端恢复点验证通过且端点匹配时才显示可用的回滚按钮。

## 9. 统一任务记录

本地记录包含：

- 任务 ID、来源、标题和创建时间。
- 精确端点标识，不包含认证凭据。
- AI 计划及用户确认时间。
- 每个步骤的命令摘要、风险、状态、耗时和脱敏输出。
- 远端恢复点路径、校验摘要、回滚可用性和过期状态。
- 执行、验证、保留和回滚事件。

记录写入现有应用数据目录中的独立 SQLite/持久化表，由主进程负责，不继续把完整任务记录放在浏览器 `localStorage`。旧的快捷命令安全记录在读取时合并显示，不强制重写旧数据。

原始输出每步设定上限；完整终端日志继续走现有工具日志和会话日志能力。导出审计报告支持 Markdown 和 JSON，并复用现有脱敏逻辑。

## 10. 安全操作中心

安全操作中心调整为四个页签：

- **执行中：** 显示计划进度、当前步骤、取消按钮。
- **可回滚：** 显示恢复点校验状态、“立即回滚”和“保留修改”。
- **历史记录：** 显示完成、失败、取消和部分完成任务。
- **旧版记录：** 兼容现有 `/tmp/shellpilot-rollback` 快捷命令记录。

“立即回滚”是正式按钮，不要求用户复制命令。点击后先重新连接或匹配原会话，再运行远端 `verify.sh`；验证失败时禁止回滚并显示原因。

## 11. 断线与故障处理

- SSH 断线：任务进入 `partially-completed`，保留恢复点；重连同一端点后允许继续验证或回滚。
- 客户端退出：执行中的任务写入最后状态；重新启动后不自动继续修改，只提示用户处理。
- 恢复点损坏：标记为不可回滚，不执行残缺脚本。
- 目标已再次变化：摘要或元数据冲突时停止自动恢复，要求用户人工确认。
- 自动网络回滚：由远端后台保护进程执行，不依赖客户端保持在线；用户确认保留后取消保护。
- 并发修改：同一端点和目标只允许一个写事务，其他任务排队或取消。

## 12. UI 入口

- 服务器状态异常项：`AI 诊断`。
- AI 助手：结构化计划卡片和任务进度卡片。
- 顶部安全中心：显示执行中和可回滚数量。
- 普通 SSH 终端：仅在识别到修改或高风险命令时弹出紧凑确认面板。
- SFTP 与内置编辑器：保存、覆盖和删除前创建文件恢复点。

所有新界面使用中文，沿用 ShellPilot 现有紧凑工具型视觉，不在聊天记录区域插入大型固定面板。

## 13. 首期交付顺序

1. 统一事务模型、持久化、端点守卫和脱敏审计。
2. 接入现有快捷命令及安全操作中心，替换分散的记录写入逻辑。
3. 接入普通终端的只读/修改/未知命令分类和安全确认。
4. 接入服务器状态异常诊断及结构化只读计划。
5. 实现计划执行、实时进度、取消和报告。
6. 接入受支持修复步骤、恢复点验证和按钮回滚。
7. 接入 SFTP/内置编辑器文件修改与删除恢复点。
8. 完成真实服务器回归后发布 0.4.0。

## 14. 验收标准

- 普通只读 SSH 命令不出现额外确认。
- 已支持修改命令在恢复点创建失败时绝不执行。
- 未知或不可逆命令不显示“可回滚”。
- AI 计划必须由用户确认，修改步骤必须再次确认。
- 任务可取消，取消后当前远端命令流被关闭且后续步骤不再执行。
- 客户端重启后仍能查看未完成和可回滚记录。
- 回滚按钮必须匹配原端点并先验证恢复点。
- 文件、权限、服务、防火墙、网络和容器状态至少各有一条自动化回滚测试。
- 真实服务器回归覆盖只读诊断、取消、文件恢复、服务恢复和断线后重连回滚。
- 代码和导出报告不包含测试服务器密码或模型 API Key。

## 15. 非目标

- 不安装服务器常驻 Agent 或守护进程。
- 不对任意脚本、数据库事务、包升级或交互式编辑器承诺自动恢复。
- 不允许 AI 绕过用户确认直接执行修改。
- 不在首期实现多服务器并行自动修复。
- 不把 ShellPilot 定位为备份软件；恢复点只服务于受控运维操作。
